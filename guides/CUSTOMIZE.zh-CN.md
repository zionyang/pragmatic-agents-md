# 裁剪指南

本指南说明如何裁剪模板，避免把用户偏好伪装成项目事实，或在无意中削弱安全边界。它负责准备候选；[安装指南](INSTALL.zh-CN.md)是写入、确认和回读流程的权威来源。

已发布模板是规则基线。不得因为更短或看似更方便，就用新的解释替代原规则。规则意图或迁移边界不清楚时，使用[规则说明](../GUIDE.zh-CN.md)。

## 先收集证据

编辑候选前，读取适用模板、既有指令、请求的目标范围和已验证项目材料。每一项可能加入的内容都归入以下三类之一：

| 类别 | 默认处理 | 示例 |
| --- | --- | --- |
| 核心安全边界 | 默认保留。弱化、删除或改变触发条件前，说明风险并取得明确确认。 | 不编造、敏感信息保护、删除确认、证据驱动排查、结果验证和针对该次操作的授权。 |
| 用户偏好 | 仅在用户明确说明或已有适用规则提供证据时本地化；不得写成项目事实。 | 用户可见语言、技术解释密度、沟通风格、输入容错和默认提交语言。 |
| 环境与授权策略 | 由真实 Agent、项目和当前用户授权共同决定；不得把旧授权或一般授权当作持续许可。 | 写入、覆盖、安装、联网、删除、提交、推送、部署、发布和中断操作。 |

## 核心安全边界

除非用户在风险已说明后明确选择变更，否则下列行为必须保留：

- 不编造 API、配置、命令、路径、目录职责、部署流程或 Agent 配置位置；
- 不暴露或写入密钥、token、密码、cookie 或环境变量值；
- 保留模板规定的删除和外部操作确认；
- 基于证据排查并定位根因，而不是掩盖失败；
- 修改后检查实际结果，并执行与风险相称的验证；
- 将 Git、发布、部署和中断操作视为独立授权。

解释可以让这些边界更易执行，但不得加入未经验证的例外、扩大 Agent 自主权限，或把针对该次操作的确认缩减为一次性偏好。

## 用户偏好

只有证据支持时，偏好才可本地化。必须保留“偏好”和“项目事实”的差异。例如，用户偏好的沟通语言属于全局规则或明确标注的偏好，不属于项目命令表；仓库已记录的发布流程是项目事实，不是个人偏好。

对于全局模板语言对，默认只允许下列已发布的本地化差异：用户可见语言、技术术语表达和默认 Git 提交信息语言。其他触发条件、必需行为、确认边界和验证期望都必须保持等义。

## 环境与授权策略

适用指令可以允许只读发现和本地验证，但候选不得静默赋予写入或外部操作权限。必须说明每项所请求操作的影响，并等待用户对该次操作作出明确确认。

不得为未知 Agent 创建“已支持”的策略。产品的配置入口或加载行为尚未验证时，结果必须保持为候选，并说明限制。

## 提问还是发现

可从可见来源确定的信息，应自行只读发现。只有用户独占、会实质改变结果或授予高风险权限的信息才提问。

| 应自行只读发现 | 必要时询问用户 |
| --- | --- |
| 既有指令文件、可见层级和目标文件是否存在。 | 请求未说明时，目标是全局、项目根目录、指定子目录还是仅候选。 |
| 现有 Agent 配置或已验证官方文档。 | 既有规则文件应保留、合并还是替换。 |
| README、贡献说明、构建配置、CI、Git 状态和已记录项目命令。 | 没有可靠指令时，偏好的语言、受众或提交惯例。 |
| 已适用的偏好和已验证项目事实。 | 希望削弱核心安全边界，或改变确认策略。 |
| 命令、目录职责、兼容性承诺或部署限制的证据。 | 可见指令的冲突无法按作用范围化解。 |

不得为了裁剪模板而要求用户披露密钥；检查中发现的敏感内容也不得输出。

## 候选审阅清单

在交给[安装指南](INSTALL.zh-CN.md)前：

1. 将每项新增内容标为已验证项目事实、用户偏好或待确认。
2. 删除猜测出的命令、路径、目录职责、配置位置和产品支持声明。
3. 将变更过的核心边界与源模板比较，并说明风险。
4. 说明选用语言和每项非默认偏好的来源。
5. 将候选关联到适用的验证与回滚方案。
