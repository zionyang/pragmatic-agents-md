# 全局规则说明

本指南说明中文全局模板中每条规则的意图、边界和迁移关注点。规范性正文仍是[templates/global/AGENTS.zh-CN.md](templates/global/AGENTS.zh-CN.md)。本指南不创建另一份模板、不赋予新权限，也不削弱模板规则。

英文对应说明见 [GUIDE.md](GUIDE.md)。已发布的[模板索引](templates/README.zh-CN.md#global-rule-mapping-and-localization-review)记录了中英文必须保持的一对一映射。

## 使用方法

审阅候选或迁移既有规则文件时，使用下表。必须保留源规则的触发条件、必需行为、风险级别、确认边界和验证要求。说明与模板似乎不一致时，应报告差异，并以模板作为规范性来源，直至文档被纠正。

## 设计原则与使用方式

### 保持全局层精简

全局指令会跨项目、跨对话持续生效。内容过长会占用上下文，也更容易造成规则重复、相互冲突和重点稀释。因此，全局层只保留跨项目高频且不可妥协的规则。

按主题分组，方便人和模型快速找到相关边界。分组标题只用于导航，不计入规则数量，也不表示优先级；真正的约束仍以每个标签附带的行为边界为准。

### 判断规则质量

不要把目标定成字数最少，也不要试图穷举所有边界。应以“最小但充分”为准。新增、合并或删减规则时，可以用下面的判断式检查：

```text
规则质量
= 必要边界清晰
+ 职责不重叠
+ 没有矛盾
+ 能改变实际行为
- 重复说明
- 假想边界
- 不必要的过程控制
```

以下情况会给规则增加不必要的负担：多条规则都要求先计划、先澄清、先确认；低风险任务也必须走完整流程；一边要求主动完成，一边又要求每个动作都先确认；已有充分证据后仍要求继续查证；或者把简单任务变成正式的设计、文档和测试流程。这些问题通常来自重复、冲突和不必要的过程控制，而不只是字数太多。新增规则前，应先确认它是否解决了反复出现的问题，能否合并到现有规则中，以及是否会给简单任务增加额外流程。

### 把规则放在合适的层级

- **全局指令**：语言和表达习惯、文件删除与高风险操作边界、Git 默认行为、排查和验证要求，以及外部操作授权边界。
- **项目级 `AGENTS.md`**：项目结构和关键入口、技术栈与依赖管理、构建/测试/检查命令、命名/风格/交付规范，以及项目特有的禁止修改区域。
- **当前对话**：本次任务的目标、临时格式或输出要求、仅对本次有效的限制（例如只分析不修改），以及本次任务的验收标准。

### 维护规则集

- 同类问题反复出现后，再考虑新增全局规则。
- 说明具体行为，不使用“认真处理”“确保高质量”之类的空泛标准。
- 新增前先检查能否合并到现有规则，避免同义重复。
- 定期删除已经失效、极少使用或应当属于具体项目的规则。
- 复杂、步骤固定且会重复的流程应制作成 Skill，不要把每一步都写入全局指令。

## 本地化审校

英文和简体中文全局模板拥有相同的四个分组和 23 条有序规则。只允许下列已发布差异本地化：

- 默认用户可见语言；
- 技术术语表达；
- 默认 Git 提交信息语言，同时 Conventional Commits 类型保持英文。

对其余每条规则，逐行比较两份模板和两份说明，确认任一版本都没有遗漏触发条件、把必需动作改为建议、扩大 Agent 权限、缩小确认边界或丢失验证要求。

## 迁移边界

从既有规则文件迁移时，先生成候选和 diff。除非用户在风险已说明后明确选择变更，否则保留已知安全边界。未知项目事实保持待确认，不得根据惯例填充。实际写入确认遵循[guides/INSTALL.zh-CN.md](guides/INSTALL.zh-CN.md)，裁剪选择遵循[guides/CUSTOMIZE.zh-CN.md](guides/CUSTOMIZE.zh-CN.md)。
